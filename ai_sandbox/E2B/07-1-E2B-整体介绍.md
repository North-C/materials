# E2B 整体介绍

## 1. 项目定位

E2B 是面向 AI Agent / Code Interpreter 场景的云端沙箱基础设施。它的目标不是提供一台长期驻留的通用虚机，而是提供一个可以被程序快速创建、连接、暂停、恢复和销毁的受控执行环境。

在这个模型里，调用方通常不是人，而是 SDK、LLM Agent 或上层应用服务。因此 E2B 的设计重点集中在三件事上：

- 安全隔离：把用户代码限制在独立的 Firecracker microVM 中执行
- 启动速度：尽量把环境准备前置到模板构建阶段，并利用快照缩短启动路径
- 编程接口：通过 SDK、HTTP API 和沙箱内 `envd` 控制面，把“创建沙箱、执行命令、读写文件、访问网络”统一成稳定接口

## 2. 整体架构

从调用路径看，E2B 可以分成四层：

1. AI 应用层
- LLM、Agent、Code Interpreter、工作流编排器

2. 接入层
- Python / JS SDK
- `POST /sandboxes`、`POST /sandboxes/{id}/connect`、`POST /sandboxes/{id}/pause` 等 API

3. 控制面
- 模板管理
- Sandbox 生命周期管理
- 资源分配与网络策略下发

4. 执行面
- Firecracker microVM
- Guest OS
- `envd` 沙箱控制器
- 用户进程与运行时

可以把它理解成一条连续链路：

`SDK 请求 -> E2B API -> 控制面编排 -> Firecracker Sandbox -> envd -> 用户命令/文件操作`

## 3. 核心能力

### 3.1 流式命令执行

E2B 的命令执行不是一次性 RPC，而是事件流模型。`Process.Start` / `Process.Connect` 会持续返回 `start`、`data`、`end`、`keepalive` 事件，适合长任务、交互式命令和 LLM 实时观察输出。

这意味着 E2B 更像“可编程的远程终端”而不是“仅返回结果的函数执行器”。

### 3.2 文件系统能力

沙箱内文件系统通过 `envd` 暴露为统一接口，支持：

- 文件读写
- 目录创建、删除、重命名
- 元数据查询
- 目录监控和事件流

这让 Agent 可以在沙箱里形成真正的工作目录，而不是只能执行单条命令。

### 3.3 模板系统

模板是 E2B 的环境制品。它把依赖安装、文件复制、启动命令、ready 检查等准备工作从“实例创建时”提前到“模板构建时”。

模板能力既支持 `e2b.Dockerfile + e2b.toml`，也支持 Template SDK 生成的结构化构建描述。模板构建结果会携带：

- 基础镜像或上游模板
- 构建步骤 `steps`
- 启动命令 `startCmd`
- 就绪检查 `readyCmd`

这使模板不仅描述“镜像内容”，也描述“沙箱启动语义”。

### 3.4 生命周期管理

Sandbox 生命周期并不只有创建和销毁，还包含：

- `running`
- `paused`
- `killed / expired`

其中 `pause` / `connect` 很关键。`connect` 在 paused 场景下会隐式恢复沙箱，因此对调用方来说，“重新连接一个沙箱”就是“继续使用之前的执行上下文”。

### 3.5 网络控制

E2B 把网络策略作为创建参数的一部分，而不是让 Guest 自己决定。创建请求里可以附带：

- `allow_out`
- `deny_out`
- `allow_public_traffic`
- `mask_request_host`

这样控制面可以在宿主机或平台代理侧统一落实出站和入站限制。

## 4. 性能设计

E2B 的性能优化核心是“把慢操作前置”：

- 环境准备前置到模板构建阶段
- 启动语义前置到 `startCmd` / `readyCmd`
- VM 初始化结果前置到快照

基于 `infra` 仓库中的真实实现，模板构建阶段会冷启动 build sandbox 并产出 build 产物；而用户创建普通 sandbox 时，节点侧通常直接从该 build 的 snapshot 恢复，再对 `envd` 做运行时初始化。

这样创建沙箱时就不必重复完整的 OS 初始化和依赖安装流程，而是优先恢复一个已经接近就绪态的执行环境。

## 5. 安全设计

E2B 的安全边界由多层组成：

- Firecracker + KVM 提供硬件级隔离
- 独立 rootfs 和 Guest OS 进程空间隔离执行上下文
- 网络策略限制出站/入站访问
- `secure=true` 时，SDK 与沙箱内 `envd` 通过 access token 通信
- 平台可进一步通过 traffic token 管理外部访问暴露的服务

这一点很重要：E2B 并不把安全建立在“用户代码遵守规则”上，而是建立在控制面和宿主机强制执行的边界上。

## 6. 典型适用场景

E2B 适合以下任务：

- LLM Code Interpreter
- AI Agent 工具调用
- 带文件上下文的代码生成与运行
- 数据分析、脚本执行、图表生成
- 需要隔离执行第三方或用户生成代码的服务

它不只是“远程执行 Python”，而是“给 Agent 一个受控、可编程、可恢复的 Linux 工作空间”。
