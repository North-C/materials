# missing evidence for failure real

This seed still lacks one decisive evidence set:

> a single set of artifacts that simultaneously proves:
> 1. control-plane restore/update request succeeded
> 2. backend/worker path progressed at least part of the way
> 3. guest-visible state still failed to converge

## What already exists in the repository

1. A success baseline `real`:
   - `cubesandbox-guest-visible-restore-baseline-real-20260622`
2. A control-plane failure `real`:
   - `cubesandbox-rollback-sandbox-not-running-real-20260622`

These are useful contrasts, but neither one proves a guest-visible convergence failure.

## Why current evidence is insufficient

Current workspace can already prove:

- a successful guest-visible convergence baseline
- a control-plane lifecycle failure

But it still cannot prove:

- that a restore/update request succeeded,
- while backend/worker advanced,
- and guest-visible convergence still failed.

Without all three, this seed must remain a `seed`, not be upgraded to `real`.

## Minimal success/failure sequence to prove next

The most valuable next evidence set would show this order:

1. `VmRestore` or `VmResumeFromSnapshot` request accepted
2. `vm ready, vsock is listening (...)`
3. `FsEvent` or equivalent worker wakeup evidence appears
4. guest agent logs `wait a pci` or enters network/storage convergence path
5. guest agent then fails with `Failed to update interface/routes/...` or equivalent

Only after a sequence like this exists should the sample be upgraded to a
guest-visible failure `real`.
